# MVPD集成功能

>[!NOTE]
>
>此页面上的内容仅供参考。 使用此API需要获得Adobe的当前许可证。 不允许未经授权使用。

## 概述 {#mvpd-int-features-overview}

Adobe Primetime身份验证支持TV Everywhere的新兴标准。 它符合 **CableLabs OLCA（在线内容访问）规范**，为从在线来源向付费电视客户交付视频提供技术要求和架构。 Adobe于2011年6月参与了CableLabs集成测试项目，并通过了服务提供商实施的测试过程。  Adobe也是 **OATC（开放式身份验证技术联盟）** 并作为该机构的一部分，参与了小组委员会的若干规范起草项目。

在介绍了Primetime身份验证OLCA合规性以及Adobe参与OATC之后，还请务必注意，Primetime身份验证实际上“与协议无关”。  但在TVE时代的这个阶段，Primetime认证的确以OLCA标准为导向。  该标准为不同的TVE参与者（程序员、MVPD和服务提供商）制定了实施TVE功能的当前商定方法。 下表列出了其中许多功能，其中包含指向相关页面的链接，这些页面提供了有关如何实施这些功能的详细信息和示例。

表中的信息旨在推动Primetime身份验证集成过程在所有MVPD集成中实现一致的功能。 “优先级”列将功能排名为A、B和C:

* 答：这些是用于初始推出集成的“必须具备”的功能。
* B — 这些是对初始集成的重要增强，将在初始推出后添加。
* C — 这些是对集成的进一步增强，可在满足“B”要求后实施。


## 1.核心功能特点 {#core-func-features}


| 不。 | 功能 | 描述 | 优先级 | 注释 |
|------|----------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------|---------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1.1 | [程序员启动的登录](/help/authentication/authn-oauth2-protocol.md) | 查看器选择MVPD并启动来自程序员品牌网站或应用程序的身份验证(AuthN)流。 | A+ |  |
| 1.2 | [基于渠道的授权](/help/authentication/authz-usecase.md) | 验证后，授权(AuthZ)便可在后台进行，只会将网络通道标识符和用户标识符传递到MVPD。 | A+ |  |
| 1.3 | 用户ID持久性 | MVPD提供模糊处理的永久用户ID。 | A |  |
| 1.4 | [单点登录支持](/help/authentication/sso-support.md) | 查看者使用MVPD在一个品牌的网站上登录，然后可以转到另一个品牌，且不会被提示再次登录。 AuthN会话在品牌之间共享。 SSO支持适用于网站和移动/设备应用程序。  对于MVPD，它需要返回用户ID或可用于跨品牌进行身份验证的其他用户令牌。 | A |  |
| 1.5 | 设备优化的登录用户体验(UX) | MVPD支持调整登录屏幕的大小，以便其符合视图所使用设备的尺寸。 | A |  |
| 1.6 | 默认MVPD选取器徽标 | MVPD提供一个URL，指向相应尺寸（112x33像素）的默认徽标。 | A | 徽标应由MVPD托管，并应由CDN缓存。 |
| 1.7 | [服务提供商范围](/help/authentication/serv-provider-scoping.md) | MVPD支持在AuthN请求中传递品牌标识符（请求者值）。 | A- | 这可以启用特定于服务提供商的登录体验。 |
| 1.8 | [多渠道授权](/help/authentication/mvpd-preflight-authz.md#preflight-multich-authz) | MVPD支持程序员在单个授权请求中发送多个信道。 | B+ |  |
| 1.9 | 基于iFrame或“JS弹出窗口”的身份验证 | 程序员能够将登录流集成到iFrame或弹出式体验中，而不是HTTP重定向中。 | B |  |
| 1.10 | [程序员启动的注销](/help/authentication/usecase-mvpd-logout.md) | 查看器在程序员网站或应用程序上以及与MVPD之间均具有经过验证的会话。 查看者可以启动联合注销程序员网站，并且注销还会清除MVPD门户上的会话。 | B | 从程序员的角度确保共享计算机受到更多保护，以免被误用。 |
| 1.11 | [自定义授权错误消息](/help/authentication/error-reporting.md) | MVPD会传递其自己的自定义错误字符串，该字符串适合程序员的联合网站或应用程序向用户显示。 | B | 启用追加销售方案 |
| 1.12 | **验证响应中的用户元数据** | MVPD AuthN响应可以包含用户元数据，该用户元数据在授权流程中充当用户体验个性化的提示。 此要求允许从MVPD向程序员提供家长控制提示。 | B- |  |
| 1.13 | MVPD启动的身份验证 | 查看器在MVPD门户上完成成功的AuthN会话，然后导航到程序员的TVE网站。 系统未提示用户输入MVPD选取器，且用户已自动进行身份验证。 | B- |  |
| 1.14 | 用户ID范围 | MVPD用户ID应以两种形式出现：一种是范围限于程序员，另一种是范围广泛的Adobe，用于欺诈。  这允许Adobe共享程序员范围的MVPD用户ID，而无需进一步加密/模糊处理。 | C |  |
| 1.15 | 基于资产的授权 | 完成AuthN后，可以通过根据需要传递可包含网络、节目、资产、家长控制评分等内容的结构化数据，在后台发生AuthZ。 这样，就可以对程序员到MVPD的每次AuthZ调用进行家长控制。 | C |  |
| 1.16 | MVPD启动的注销 | 查看者在程序员网站或应用程序上以及与MVPD之间均具有经过验证的会话。 查看器可以启动联合注销MVPD站点，该联合注销也会清除所有联合程序员站点上的会话。 | C |  |
| 1.17 | 授权义务 | MVPD在AuthZ响应中提供了其他条件，如日志记录或更新的家长控制最高评分(OLCA)。 | C |  |
| 1.18 | IP地址上下文 | MVPD要求明确传递IP地址。 对于AuthZ（调用位于服务器端），这会提供有关用户在AuthZ调用中来自何处的MVPD欺诈跟踪信息。 | C |  |
| 1.19 | 动态定义的令牌持久性生存时间(TTL)值 | MVPD能够通过响应中的属性动态设置Primetime身份验证令牌的TTL，以便Adobe不会出现TTL更改的循环。 | C- |  |
| 1.20 | 设备类型 | MVPD支持在AuthN或AuthZ请求中传递设备类型。 此属性会告知MVPD将在其中使用内容的设备的性质，以便MVPD可以调整AuthN或AuthZ令牌的TTL，以符合其自身对设备的安全注意事项。 | C- | 这对于无客户端平台非常有用，在该平台中，在Primetime身份验证端，可能很难将每种设备类型显示为配置设置。 |
